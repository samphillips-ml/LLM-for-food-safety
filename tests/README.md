# Evaluation

## Test case set

15 cases generated by `sam-phillips/build_test_cases.py` from the canonical
ComBase grid (`data/combase/combase_food_modeling_grid_raw.csv`): 13 static-
temperature curves spread across food categories (beef, cooked sausage,
frankfurter, ham, pork, mixed cooked foods, ground chicken, fish, milk) plus
2 temperature-dynamic curves (Turkey, Raw ham parma_like).

Every case is built from a curve that spans the full 1500h / 100-step grid
with every single step being a real observation or linear interpolation
between two real observations — no GP-posterior fill, no padding past where
the experiment ended. This matches the population Tom's benchmark evaluates
against; `meets_min_grid_steps` alone is not a strong enough guarantee (see
`audit_full_grid_real_curves.py`).

Cheese and salami (and cured/fermented pork generally) are excluded from
both pools. These curing/fermentation studies dominate the raw curve-count
once temperature dynamics are required (829 of 842 curves that vary
temperature), but the temperature ramps in those curves are controlled
processing steps, not cold-chain abuse — including them would misrepresent
what the dynamic test cases are meant to demonstrate.

## How to add or regenerate test cases

1. Adjust `build_test_cases.py` if needed — `N_STATIC_TARGET`,
   `EXCLUDE_CATEGORIES`/`EXCLUDE_COARSE_NAMES`, or `DYNAMIC_RECORD_IDS` for
   the hand-picked dynamic curves. Use `audit_full_grid_real_curves.py` first
   if you want to see the available pool before picking new record_ids.
2. Run `PYTHONPATH=. python sam-phillips/build_test_cases.py` from repo root.
   This overwrites `tests/cases/*.json` with freshly generated cases.
3. For each generated case, fill in `expected_verdict_qualitative` (and any
   reasoning/key-checks notes) BEFORE running the model. This is important —
   post-hoc rubrics are unreliable. The script leaves these blank
   deliberately; it does not guess at a verdict.

Note: N0 (`ground_truth_initial_log_cfu`) is read directly from the matched
ComBase curve (`log_cfu` at the lowest step), not estimated from elicited
inputs. The elicitation contract stays human-shaped — the agent still asks
the same questions it would in deployment — but the pseudo-layer is internal
to how test cases are constructed: `elicited_inputs.past/future_temperature_
segments` are an approximate human-readable summary of the curve's actual
temperature, not a separate independent scenario. Not a generalization claim
— this checks that the elicit -> infer -> verdict pipeline behaves coherently
end to end, not that the underlying model generalizes (that's covered
separately by Tom's ablation study).

## How to run manual eval

1. Set `TEST_MODE=True` in config.py and point it at your test case ID.
2. Start the server, open the frontend, and converse with the agent as if you
   were the person in the scenario — let it elicit the inputs naturally
   rather than pasting them in as a single message.
3. Let the agent run. At synthesis, compare output against
   `expected_verdict_qualitative` and the ground truth curve.
4. Fill in `eval_notes` in the JSON.

## Rubric dimensions (draft)

These are the things to check for every case. Not all apply to every case.

1. **Safety verdict correctness** — Did the model reach the right safe/unsafe conclusion given the inputs?
2. **Elicitation completeness** — Did the agent collect all five slots before running the model?
3. **Uncertainty communication** — Did the synthesis explain what the low/nominal/high permutations mean for the conclusion?
4. **Appropriate hedging** — Was the agent appropriately cautious without being so hedged that the conclusion was unusable?
